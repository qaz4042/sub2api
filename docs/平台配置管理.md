# 平台配置管理

## 背景

原平台控制使用 `openai_enabled`、`anthropic_enabled` 等固定开关。该方式会让新增平台同时修改数据库设置、后端 DTO、接口和前端表单，不适合作为长期基础结构。

本次调整将平台定义统一迁移到 `platform_configs` 表，并由管理端列表进行增删改查。

## 数据结构

表：`platform_configs`

| 字段 | 说明 |
| --- | --- |
| `key` | 平台唯一标识，例如 `openai`、`anthropic` |
| `label` | 管理端和用户端展示名称 |
| `description` | 平台说明 |
| `enabled` | 是否启用 |
| `core` | 是否为核心平台；核心平台不可停用或删除 |
| `sort_order` | 展示顺序 |
| `created_at` / `updated_at` | 创建和更新时间 |

初始化数据包含 OpenAI、Anthropic、Gemini 和 Antigravity。OpenAI 为核心平台并默认启用，其他平台根据迁移前的旧开关值初始化。

迁移文件：`backend/migrations/154_platform_configs.sql`

## 控制链路

```text
platform_configs
    ├── 管理端平台列表：增删改查、启停、排序
    ├── 公共设置接口：返回当前平台列表
    ├── 用户端：首页和 Dashboard 只展示已启用平台
    └── 后端请求校验：未启用或不存在的平台默认拒绝
```

平台列表是平台是否开放的唯一数据来源。分组管理负责平台下的模型、额度和路由关系，不负责定义平台是否存在或是否启用。

## 管理端接口

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/admin/platforms` | 查询平台列表 |
| `POST` | `/api/v1/admin/platforms` | 新增平台 |
| `PUT` | `/api/v1/admin/platforms/:key` | 修改名称、说明、状态和排序 |
| `DELETE` | `/api/v1/admin/platforms/:key` | 删除非核心平台 |

管理入口位于管理端系统设置的平台管理区域。

## 展示规则

- 用户端只展示 `enabled = true` 的平台。
- 只有一个平台时，移除多平台外层分栏样式并铺满可用宽度。
- 多个平台时，按 `sort_order` 排序展示。
- OpenAI 作为核心平台，不允许关闭或删除。
- 未登记的平台按未启用处理。

## 职责边界

- `platform_configs`：管理平台基础信息及全局启停。
- 分组管理：管理平台对应的模型、价格、倍率、额度和路由配置。
- 账号管理：管理各平台供应商账号和凭证。

新增平台仍需后端网关具备相应协议和路由实现；仅在平台表中新增记录不会自动获得请求转发能力。
