# 运营前配置与数据检查清单

检查日期：2026-06-21  
适用项目：Sub2API  
当前实例：`portal.lizubin.online`

## 一、结论

当前实例更适合内部试运行，暂不适合直接公开收费运营。

当前已经具备的基础条件：

- Sub2API、PostgreSQL、Redis、Mihomo 容器均处于健康状态。
- 应用端口只绑定到 `127.0.0.1:8080`。
- PostgreSQL 和 Redis 未直接暴露到公网。
- 服务运行在 `release` 模式。
- PostgreSQL、Redis、JWT、TOTP 和管理员密码均已配置。
- HTTPS 当前可以正常访问。
- 服务器具有约 2 GiB Swap。
- 已完成七牛云 S3 兼容对象存储配置，用于 PostgreSQL SQL 备份。

当前主要缺口：

- 七牛云异地 SQL 备份已经配置，但仍需完成自动备份、失败通知和恢复演练验收。
- URL 白名单未开启，并允许访问私网地址和 HTTP 地址。
- 管理员 TOTP 双因素认证未启用。
- 应用连接池、PostgreSQL 服务端参数和 Redis 服务端参数明显高于当前 2 GiB 小服务器的合理起步值。
- 公开注册、支付、邮件、条款、风控等功能尚未形成运营闭环。
- 站点仍使用默认品牌，客服、文档和 API 地址为空。
- 计费、额度、防刷和故障告警需要进行端到端验收。

当前已确认的运营决策：

- 正式公网入口暂定为中国内地服务器，域名为 `portal.lizubin.online`。
- 当前阶段暂时忽略 ICP 备案问题，公开运营前应重新评估入口和备案方案。
- 新用户默认余额为 `10 USD`。
- 每个上游账号最大并发为 `5`。

## 二、10 分：运营前必须完成

### 1. 建立数据备份与恢复闭环

当前状态：

- 已完成七牛云 S3 兼容对象存储配置，数据库 SQL 备份可保存到服务器之外。
- 内置备份直接写入对象存储，本地 `/opt/sub2api/backups/` 没有文件不能单独证明备份未执行。
- 数据库迁移为 forward-only，发生错误时主要依赖备份恢复。
- 尚需确认定时任务、实际备份对象、失败通知和完整恢复流程均可用。

建议：

- 在管理后台测试七牛云 S3 连接，并确认 Bucket、Endpoint、Region、Prefix 和凭证配置正确。
- 每日自动执行 PostgreSQL 完整 SQL 备份。
- 备份 `.env`、Compose 配置和必要的应用数据目录。
- 至少保留最近 7 份日备份和 4 份周备份。
- 对象存储凭证、数据库密码、加密密钥和恢复文档应离线保存，不能只存在当前服务器数据库中。
- 对七牛云 Bucket 配置最小权限、访问控制和生命周期策略。
- 在临时 PostgreSQL 实例中执行一次完整恢复。
- 目标恢复点 RPO 建议不超过 24 小时，目标恢复时间 RTO 建议不超过 2 小时。

验收标准：

- 管理后台七牛云 S3 连接测试通过。
- 备份任务可以自动执行，并在七牛云 Bucket 中产生非空、可下载的备份对象。
- 连续检查至少 3 次定时备份均按计划完成，保留和清理策略符合预期。
- 备份失败时管理员能够收到通知。
- 临时库恢复后，用户、账号、分组、API Key 和使用记录数量符合预期。
- 恢复后的关键账号凭证、系统设置和计费数据可正常读取。
- 实际恢复时间和可接受的数据丢失窗口满足 RTO、RPO。
- 恢复步骤已经记录在运维文档中。

### 2. 收紧 URL 和 SSRF 安全配置

当前状态：

```env
SECURITY_URL_ALLOWLIST_ENABLED=false
SECURITY_URL_ALLOWLIST_ALLOW_INSECURE_HTTP=true
SECURITY_URL_ALLOWLIST_ALLOW_PRIVATE_HOSTS=true
```

建议：

```env
SECURITY_URL_ALLOWLIST_ENABLED=true
SECURITY_URL_ALLOWLIST_ALLOW_INSECURE_HTTP=false
SECURITY_URL_ALLOWLIST_ALLOW_PRIVATE_HOSTS=false
```

- `SECURITY_URL_ALLOWLIST_UPSTREAM_HOSTS` 只填写实际使用的上游域名。
- 如果确实需要连接内部 CRS 或代理服务，应单独列入白名单，不应全局放开私网。
- 保持代理初始化失败时禁止回退直连，避免服务器出口 IP 泄露。

验收标准：

- 已配置的合法上游可以正常调用。
- `http://` 上游地址被拒绝。
- `127.0.0.1`、链路本地地址和内网地址被拒绝。
- 未列入白名单的公网域名被拒绝。

### 3. 启用管理员双因素认证

当前状态：

- TOTP 加密密钥已经固定配置。
- 系统 TOTP 功能尚未启用。

建议：

- 在管理后台启用 TOTP 功能。
- 管理员账号立即绑定 TOTP。
- 恢复码离线保存，不和服务器密码存放在同一位置。
- 不共享管理员账号；需要多人运维时创建独立账号。

验收标准：

- 仅有密码不能完成管理员登录。
- TOTP 正常登录和恢复流程均经过测试。

### 4. 降低小服务器的连接池和并发配置

当前服务器：

- 2 vCPU。
- 约 1.8 GiB 内存。
- 2 GiB Swap。

当前配置：

```env
DATABASE_MAX_OPEN_CONNS=256
DATABASE_MAX_IDLE_CONNS=128
REDIS_POOL_SIZE=4096
REDIS_MIN_IDLE_CONNS=256
GATEWAY_IMAGE_CONCURRENCY_ENABLED=false
GATEWAY_IMAGE_CONCURRENCY_MAX_CONCURRENT_REQUESTS=0
```

建议起步值：

```env
DATABASE_MAX_OPEN_CONNS=30
DATABASE_MAX_IDLE_CONNS=5
REDIS_POOL_SIZE=64
REDIS_MIN_IDLE_CONNS=4
GOMEMLIMIT=600MiB
GATEWAY_IMAGE_CONCURRENCY_ENABLED=true
GATEWAY_IMAGE_CONCURRENCY_MAX_CONCURRENT_REQUESTS=1
GATEWAY_IMAGE_CONCURRENCY_OVERFLOW_MODE=reject
GATEWAY_IMAGE_CONCURRENCY_MAX_WAITING_REQUESTS=0
```

同时建议：

- 为 Sub2API、PostgreSQL、Redis 和 Mihomo 设置合理的容器内存限制。
- 将 PostgreSQL `max_connections` 起步值调整到约 80，`shared_buffers` 调整到约 128 MiB，并限制 `work_mem`。
- 为 Redis 设置合理的 `maxclients`、`maxmemory` 和内存淘汰策略。
- 首期通过分组模型范围和模型映射禁用图片模型；图片并发参数只是资源保护，不等于关闭图片功能。
- 不直接套用面向数千并发用户的默认连接池参数。

验收标准：

- 通过容器环境和服务端查询确认参数已经实际生效，而不只检查 `.env` 文件。
- 连续压测 30 分钟无 OOM。
- Swap 不持续快速增长。
- PostgreSQL 不出现连接耗尽。
- Redis 不出现连接池等待超时。
- 停止压测后内存能够回落到稳定区间。

### 5. 固化公网入口方案

当前状态：

- 域名曾出现阿里云中国内地入口的 ICP 备案拦截。
- 正式公网入口已暂定为中国内地服务器。
- 正式域名为 `portal.lizubin.online`。
- 当前阶段暂时忽略 ICP 备案问题。

后续事项：

- 公开运营前重新评估 ICP 备案及阿里云接入备案要求。
- 如果中国内地入口持续出现备案拦截，再评估迁移至中国香港或其他境外入口。

验收标准：

- 使用国内不同运营商网络和境外网络测试 HTTPS。
- 普通 API 请求、SSE 长连接和 WebSocket 均可完成。
- HTTP 正确跳转 HTTPS。
- 证书自动续期 dry-run 通过。
- 外部请求能够进入 Nginx 和应用日志。

### 6. 保持封闭注册，先完成小规模试运行

当前公开设置：

- 注册关闭。
- Backend Mode 未开启。
- 邮件验证和密码找回关闭。

建议：

- 第一阶段保持注册关闭。
- 由管理员创建 3–10 个测试用户。
- 封闭试运行保持 Backend Mode 关闭；该模式会禁止非管理员访问用户自助功能。
- 只有实例仅供管理员后台操作、不需要普通测试用户登录和创建 Key 时，才启用 Backend Mode。
- 在邮件、验证码、防刷和条款未配置前，不开放公网注册。

验收标准：

- 未授权访客不能注册。
- 测试用户可以正常登录、创建 Key 和调用 API。
- 管理员能够禁用用户及其所有 Key。

### 7. 验证模型定价和计费正确性

风险点：

- 模型价格可从远程价格仓库同步。
- 分组、渠道、用户和账号均可能存在倍率。
- 模型映射可能导致请求模型、上游模型和计费模型不同。

建议：

- 只开放已经核价的模型。
- 为每个开放模型记录：
  - 上游模型。
  - 计费模型。
  - 输入价格。
  - 输出价格。
  - 缓存价格。
  - 平台销售倍率。
  - 预计毛利。
- 禁用尚未确认定价或映射的模型。
- 首期不开放图片模型。
- 对已开放模型分别测试普通、流式、缓存和推理请求。

验收标准：

- 使用固定 Token 数请求进行人工计算。
- 人工结果与系统账单误差低于预设阈值。
- 重试、失败和上游切换不会重复扣费。
- 用户余额、使用记录和聚合报表能够互相核对。
- 测试矩阵至少覆盖普通成功、流式成功、缓存命中、上游 4xx、上游 5xx、超时、重试、客户端中断和余额不足。

### 8. 建立配额和防刷闭环

建议至少配置四层限制：

- 用户并发与 RPM。
- API Key 并发、RPM 和额度。
- 分组 RPM 和模型范围。
- 上游账号并发和额度。

起步原则：

- 新用户默认余额为 `10 USD`。
- 不设置“0 表示不限”的默认 RPM。
- 首批用户使用较低并发和日额度。
- API Key 必须绑定明确分组。
- 不允许余额无限透支。

验收标准：

- 达到并发限制后返回明确错误。
- 达到 RPM 限制后返回 429。
- 达到日额度或余额不足后停止调用。
- 并发槽位在请求异常中断后可以自动释放。
- 同一个请求不会被重复扣费。

### 9. 明确日志、隐私和数据保留策略

建议：

- 默认不记录完整请求正文和提示词。
- 禁止日志包含 API Key、OAuth Token、上游 Cookie 和代理密码。
- 确认上游错误体日志不会记录用户敏感内容。
- 将使用记录保留期控制在实际业务需要范围内，初期建议 30–90 天。
- 明确哪些数据用于计费、排障和审计。
- 向用户说明数据存储位置、保留时间和删除机制。

验收标准：

- 抽查正常调用、失败调用、登录和账号测试日志。
- 日志中不存在完整凭证和敏感请求内容。
- 到期数据能够按计划清理。
- 清理任务不会影响账单和订单审计。

### 10. 建立故障监控与通知闭环

当前状态：

- 渠道监控已开启。
- 邮件和运营告警尚未形成完整闭环。

建议监控：

- 应用、PostgreSQL、Redis、Mihomo 容器状态。
- HTTP 健康检查。
- 上游渠道成功率和首 Token 延迟。
- 账号余额、配额和到期时间。
- 磁盘、内存、Swap 和数据库连接。
- 证书到期时间。
- 备份任务执行结果。

验收标准：

- 人工禁用一个渠道，管理员能够收到通知。
- 人工停止一个容器，监控能够发现。
- 磁盘达到测试阈值时能够告警。
- 告警包含时间、对象、错误原因和处理入口。

### 11. 完成公开运营和上游授权合规审查

本项不阻塞 3–10 人封闭内部试运行，但属于开放注册、邀请外部用户或公开收费前的硬门槛。

至少确认：

- 运营主体、服务器所在地、目标用户所在地和实际业务开展地适用的备案、许可、税务和消费者权益要求。
- 上游账号或 API 的商业使用、账号共享、额度分发、转售、地区限制和技术限制。
- 不将个人订阅权益转换为未经授权的多用户收费 API。
- 服务条款、隐私政策、可接受使用政策、退款规则和投诉处理机制已经发布。
- 用户数据的存储位置、跨境传输、保留、删除和安全事件响应责任已经明确。
- 公开收费前由具备资质的专业人员根据实际业务模式进行合规审查。

验收标准：

- 每种上游账号均有来源、授权范围和允许使用方式的书面记录。
- 无法确认授权或转售范围的账号不用于对外服务。
- 开放注册或收费前，用户必须确认当前版本的服务条款和隐私政策。
- 在本项完成前，实例保持封闭注册、不接在线支付、不对公众宣传收费服务。

## 三、6 分：建议在首批用户前完成

### 1. 修改站点品牌和公开信息

当前状态：

- 网站名称为 `Sub2API`。
- 副标题仍为默认英文内容。
- API Base URL 为空。
- 客服信息为空。
- 文档地址为空。

建议配置：

- 网站名称和 Logo。
- 简短、准确的服务副标题。
- API Base URL。
- 使用文档地址。
- 客服邮箱或工单入口。
- 服务状态页。

### 2. 补充服务条款和隐私政策

当前状态：

- 登录协议确认关闭。
- 默认条款文档内容为空。

建议至少提供：

- 服务条款。
- 隐私政策。
- 可接受使用政策。
- 支持的国家和地区。
- 退款和余额处理规则。
- 服务中断及上游不可用说明。

本节用于完善具体文档内容；公开运营的合规硬门槛及验收要求见“二、11”。

公共运营前应启用登录或注册条款确认，并记录条款版本、确认时间和用户标识。

### 3. 隐藏未形成闭环的功能

当前状态：

- 支付关闭。
- 邀请返利关闭。
- 购买订阅关闭。
- 优惠码功能开启。

建议：

- 暂时关闭优惠码。
- 保持支付、返利和购买订阅关闭。
- 隐藏尚未使用的自定义菜单和入口。
- 每次只开放一个能够完整验收的功能。

### 4. 配置邮件验证和密码找回

如果准备开放用户自助注册：

- 配置 SMTP。
- 测试发信域名和 SPF、DKIM、DMARC。
- 同时启用邮件验证和密码找回。
- 为验证码发送设置用户级和 IP 级限制。

如果暂时不配置邮件，则继续采用管理员创建用户模式。

### 5. 调整数据保留周期

默认聚合数据可能保留较长时间，而当前磁盘只有 40 GiB。

建议分别定义：

- 原始使用记录保留天数。
- 聚合统计保留天数。
- 错误日志保留天数。
- 支付订单和财务审计保留周期。
- 内容审核记录保留周期。

同时设置磁盘 70%、80%、90% 分级告警。

### 6. 固化发布和回滚流程

每次发布应记录：

- Git commit。
- 镜像 tag。
- 镜像 digest。
- 数据库迁移列表。
- 发布前备份。
- 发布后健康检查。
- 回滚触发条件和回滚步骤。

不要直接使用会漂移的 `latest` 镜像。

### 7. 准备公告和客服模板

建议准备以下模板：

- 服务和计费说明。
- 计划维护通知。
- 上游故障通知。
- 余额或额度异常通知。
- API Key 泄露处理。
- 退款和争议处理。

### 8. 补齐上游账号运营数据

每个上游账号至少记录：

- 账号来源和负责人。
- 平台和账号类型。
- 成本和额度。
- 到期时间。
- 代理出口。
- 可用模型。
- 并发上限。
- 风险等级。
- 最近测试时间。
- 停用和替换策略。

## 四、2 分：有真实需求后再启用

以下能力不应阻塞首期内部运营：

- 开放注册及 Cloudflare Turnstile。
- 在线支付和自动退款。
- 订阅套餐自动购买。
- 邀请返利和营销优惠码。
- GitHub、Google、微信或 OIDC 登录。
- 高级 WebSocket 连接池调优。
- 调度权重和复杂模型回退策略。
- TLS 指纹和客户端模拟设置。
- 图片生成。
- 内容审核自动封禁。
- 自定义菜单和首页装修。
- 面向用户公开渠道状态。

这些功能只有在基础调用、计费、限额、告警和备份闭环稳定后再逐项启用。

## 五、推荐的最小运营闭环

```text
完成备份恢复
  → 收紧安全和资源参数
  → 配置一个分组、一个渠道和少量模型
  → 管理员创建 3 个测试用户
  → 人工发放小额余额或订阅
  → 完成至少 20 笔调用和计费核对
  → 模拟超额、上游失败、恢复和告警
  → 连续运行 7 天
  → 再决定是否开放邀请注册
```

第一阶段建议明确保持：

- 不开放注册。
- 不接在线支付。
- 不启用返利和优惠码。
- 通过分组模型权限和模型映射禁用图片模型。
- 保持 Backend Mode 关闭，使普通测试用户能够登录、创建 Key 和查看使用情况。
- 不宣传高可用 SLA。
- 不向公众宣传或提供收费服务。

第一阶段只验证：

```text
创建用户
  → 发放 API Key
  → API 调用
  → 正确计费
  → 配额限制
  → 故障告警
  → 数据备份与恢复
```

## 六、推荐执行顺序

### 闭环 1：备份恢复

目标：任何错误都可以恢复。

完成条件：

- 七牛云 S3 连接测试通过。
- 连续至少 3 次自动备份正常，异地对象可下载。
- 临时库恢复成功。
- RPO、RTO 达标，恢复所需凭证和文档可在服务器故障后独立取得。

### 闭环 2：安全收口

目标：减少公网和内部网络攻击面。

完成条件：

- URL 白名单开启。
- 私网和 HTTP 上游默认禁止。
- 管理员 TOTP 开启。
- 日志抽查通过。

### 闭环 3：资源收口

目标：确保 2 GiB 服务器在小流量下稳定。

完成条件：

- 连接池下调。
- PostgreSQL 和 Redis 服务端参数完成收口。
- 容器资源受控。
- 实际运行配置核验通过。
- 压测无 OOM 和连接耗尽。

### 闭环 4：单渠道计费

目标：证明一条最小业务链路可正确结算。

完成条件：

- 一个渠道和少量模型可用。
- 定价和倍率已人工核对。
- 失败、重试和切换不重复扣费。
- 普通、流式、缓存、失败、超时、中断和余额不足测试矩阵通过。

### 闭环 5：测试用户运营

目标：验证真实用户的完整使用过程。

完成条件：

- 3–10 名测试用户稳定运行 7 天。
- 限额、告警、禁用和恢复均测试通过。
- 常见问题有处理文档。

### 闭环 6：决定是否扩展

只有前五个闭环稳定后，再依次评估：

1. 邀请注册。
2. 邮件验证和密码找回。
3. 服务条款确认。
4. 支付和套餐。
5. 返利与营销。
