# 当前风控流程

更新日期：2026-06-26
适用对象：Sub2API 当前代码实现

## 一、定位

当前项目已有的风控主要是“内容审核风控”，不是完整的用户风险画像系统。

它解决的问题是：

- 对请求内容做审核。
- 命中风险内容时拦截或记录。
- 累计违规后自动禁用用户。
- 后台查看日志、测试审核 Key、解封用户。

## 二、整体流程

```mermaid
flowchart TD
    A[网关收到 API 请求] --> B[解析用户/API Key/分组/模型/请求体]
    B --> C[调用 checkContentModeration]
    C --> D{risk_control_enabled 是否开启}
    D -- 否 --> Z[放行]
    D -- 是 --> E[加载 content_moderation_config]
    E --> F{内容审核 enabled 且 mode 非 off}
    F -- 否 --> Z
    F -- 是 --> G{分组和模型是否在审核范围}
    G -- 否 --> Z
    G -- 是 --> H[按协议提取用户输入]
    H --> I{输入是否为空}
    I -- 是 --> Z
    I -- 否 --> J[规范化文本/图片并计算输入 hash]
    J --> K{pre_block 且命中关键词}
    K -- 是 --> L[本地拦截 keyword_block]
    K -- 否 --> M{pre_hash_check 命中 Redis hash}
    M -- 是 --> N[本地拦截 hash_block]
    M -- 否 --> O{采样是否命中}
    O -- 否 --> Z
    O -- 是 --> P{是否有可用审核 API Key}
    P -- 否 --> Z
    P -- 是 --> Q{审核模式}
    Q -- observe --> R[请求先放行，异步审核]
    Q -- pre_block --> S[同步调用 moderation API]
    S --> T{审核是否命中阈值}
    T -- 否 --> U[允许请求，可选记录 non-hit]
    T -- 是 --> V[拦截当前请求 block]
    L --> W[异步写日志和副作用]
    N --> W
    V --> W
    R --> X[Worker 调用 moderation API]
    X --> Y{异步审核是否命中}
    Y -- 是 --> W
    Y -- 否 --> U
    W --> AA[写 content_moderation_logs]
    W --> AB[记录 flagged hash]
    W --> AC[累计用户违规次数]
    AC --> AD{达到自动封禁阈值}
    AD -- 是 --> AE[users.status = disabled]
    AE --> AF[清理用户鉴权缓存]
    AD -- 否 --> AG[仅记录违规]
```

## 五、模式差异

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Gateway as 网关
    participant CM as ContentModerationService
    participant API as Moderation API
    participant DB as content_moderation_logs

    alt pre_block
        Client->>Gateway: 发送请求
        Gateway->>CM: 同步审核
        CM->>API: /v1/moderations
        API-->>CM: 审核结果
        alt 命中
            CM-->>Gateway: Block decision
            Gateway-->>Client: 4xx 拦截
            CM->>DB: 异步写日志/封禁副作用
        else 未命中
            CM-->>Gateway: Allow
            Gateway-->>Client: 转发上游结果
        end
    else observe
        Client->>Gateway: 发送请求
        Gateway->>CM: 入队异步审核
        Gateway-->>Client: 继续转发上游结果
        CM->>API: Worker 异步审核
        API-->>CM: 审核结果
        CM->>DB: 命中或按配置记录
    end
```

## 六、cyber_policy 旁路

```mermaid
flowchart TD

    B -- 否 --> C[正常结束]

    H -- 否 --> J[累计违规次数]
    J --> K{达到封禁阈值}
    K -- 是 --> L[禁用用户并清理鉴权缓存]
    K -- 否 --> I
    D --> M{cyber_session_block_enabled 是否开启}
    M -- 是 --> N[Redis 写入会话级屏蔽 TTL]
    M -- 否 --> C
```

## 八、当前边界

当前实现已经形成小闭环：

- 请求审核。
- 命中拦截或观察。
- 写审计日志。
- 累计违规。
- 自动禁用用户。
- 后台人工解封。

当前没有完整实现：

- 用户风险等级表。
- 独立风控动作表。
- 用户/IP/设备/支付账户画像。
- 按风险等级自动调整额度。
- 专门的人工复核工单流。

因此，当前机制适合先防“内容违规导致上游风险”，但还不是完整的“平台滥用风控系统”。
