# 开源更新与能力扩展手册

## 目标

持续吸收 `Wei-Shaw/sub2api` 上游更新，同时保留产品、运营和部署差异。上游核心保持可更新，定制能力独立实现，只通过少量接入点连接。

重点是减少 gateway、settings、routes、router、菜单和语言包等热点文件的重复冲突，不追求一次性完成架构重构。

## 归属判断

| 需求 | 处理方式 |
| --- | --- |
| 通用修复、协议兼容、安全与稳定性改进 | 拆成独立提交，优先贡献上游。 |
| 支付、风控、私有登录、品牌、运营或部署差异 | 放定制扩展目录。 |
| 必须进入主流程 | 先补最小 contract、registry 或 slot；私有实现不散落进核心文件。 |

不复制完整 `handler`、`service`、`views/admin` 或基础组件目录；不长期堆叠 `if privateXxx`；不修改历史 migration。

## 目录与接入

### 前端

```text
frontend/src/
  components/、views/、api/       # 上游通用实现
  extensions/private/<feature>/   # 私有页面、组件、API、状态、语言包、测试
  registries/                     # 菜单、路由、设置、页面扩展等少量入口
```

私有功能应拥有自己的组件、弹窗、状态、API、语言包和测试。公共页面只负责一次性接入：新增按钮和弹窗优先使用命名 slot 或 action/dialog registry；不在公共页面增加私有 `showXxx`、`selectedXxx`、`handleXxx`。

### 后端

```text
backend/internal/
  handler/、service/、repository/  # 上游核心流程
  contracts/                      # 稳定扩展接口
  extensions/private/<feature>/   # 私有 routes、handler、service、repository、测试
  server/extension_registry.go    # 集中装配扩展
```

网关风控使用 policy 或 middleware；支付使用 provider；私有登录和管理 API 使用 route registrar；运营配置使用 config provider。核心文件只负责调用契约或注册扩展。

### 其他资产

| 范围 | 约定 |
| --- | --- |
| 迁移 | 只追加；私有迁移使用独立命名和注册约定。 |
| 配置 | 私有变量以 `PRIVATE_` 为前缀，由私有配置模块读取。 |
| 部署、CI | 放 `deploy/private/`、`private-*.yml`，以覆盖或额外入口接入。 |
| 文档、品牌 | 放私有目录或既有 `docs/platform/`，不替换上游默认资产。 |
| 测试 | 跟随私有模块；公共接入点保留最小契约测试。 |

## 分支与提交

```text
upstream/main -> main-upstream -> private/main
                   通用基线         定制补丁栈
```

- `main-upstream` 跟随上游，只暂存待回馈的通用补丁。
- `private/main` 基于基线，只包含明确的定制能力。
- 功能从当前定制基线创建；提交保持单一意图，不混入文案、部署、格式化和无关重构。

## 一项能力的闭环

1. 归类：上游候选或私有扩展。
2. 查找接入点；缺失时仅增加一个最小 contract、registry 或 slot。
3. 在独立 feature 完成业务、配置和测试。
4. 只修改一个公共接入点。
5. 验证关闭配置后的默认行为、权限和错误路径。
6. 单独提交并注明接入点；可复用部分准备贡献上游。

既有深度耦合代码不做一次性重构。以后修改该能力时，再将新增逻辑迁出热点文件并补契约测试。

## 上游同步与验证

```bash
git fetch upstream
git switch main-upstream
git merge --ff-only upstream/main
git switch private/main
git rebase main-upstream
```

冲突处理顺序：保留上游核心与安全修复，调整私有模块适配新 DTO/契约；若仍反复冲突，再增加小接入点。不要在冲突处重新塞回私有逻辑。

按改动执行最小充分验证：

- gateway、计费、认证、迁移：相关 Go 测试。
- 页面、路由、菜单、弹窗：相关 Vitest 与 `pnpm --dir frontend run typecheck`。
- 构建、嵌入式前端：对应 build。
- 配置、部署：静态校验与文档核对；除非明确批准，不部署。

## 每周检查

- [ ] 评估上游安全、稳定性、迁移和公共 API 更新。
- [ ] 拆分可回馈的通用修复。
- [ ] 检查私有功能是否仍只修改约定接入点。
- [ ] 确认私有迁移、配置、部署模板和回滚说明一致。
- [ ] 发布前验证定制关闭时仍保持上游默认行为。
