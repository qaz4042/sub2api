# 标准风控机制

更新日期：2026-06-26
适用对象：Sub2API 自营实例

## 一、目标

用最小闭环降低劣质用户造成的平台封号、资金损失和合规风险。

核心闭环：

- 识别风险
- 限制风险
- 暂停风险
- 人工复核
- 沉淀规则

## 二、推荐流程

```mermaid
flowchart TD
    A[用户注册或创建 API Key] --> B[默认低信任等级]
    B --> C[发起 API 请求]
    C --> D[采集风险信号]
    D --> E{是否命中硬规则}
    E -- 是 --> F[直接拒绝请求]
    F --> G[写入风控事件]
    G --> H{是否达到暂停阈值}
    H -- 是 --> I[暂停用户或 API Key]
    H -- 否 --> J[保持低额度观察]
    E -- 否 --> K{是否超过用量或频率阈值}
    K -- 是 --> L[限速或降低额度]
    L --> G
    K -- 否 --> M[正常放行]
    M --> N[持续累计正常行为]
    N --> O{是否满足提额条件}
    O -- 是 --> P[提高信任等级和配额]
    O -- 否 --> C
    I --> Q[人工复核]
    Q --> R{复核结论}
    R -- 恢复 --> S[恢复账户并记录原因]
    R -- 拉黑 --> T[加入黑名单]
    R -- 继续观察 --> J
    S --> C
```

## 三、关键数据模型

较小闭环建议优先落 4 类数据，其中请求日志如已有可复用。

```mermaid

    users ||--o| risk_profiles : has
    users ||--o{ risk_events : triggers
    users ||--o{ risk_actions : receives
    users ||--o{ usage_counters : consumes

    risk_profiles {
        bigint user_id PK
        string risk_level
        string status
        string quota_tier
        string reason
        datetime updated_at
    }

    risk_events {
        bigint id PK
        bigint user_id
        string event_type
        string severity
        string source
        json metadata
        datetime created_at
    }

    risk_actions {
        bigint id PK
        bigint user_id
        string action_type
        string reason
        bigint operator_id
        datetime created_at
    }

    usage_counters {
        bigint id PK
        bigint user_id
        string window_type
        int request_count
        decimal cost_used
        datetime window_start
        datetime updated_at
    }
```

## 四、规则分层

```mermaid
flowchart LR
    A[风险信号] --> B[硬规则]
    A --> C[软规则]
    A --> D[成长规则]

    B --> B1[违法关键词]
    B --> B2[黑名单 IP/设备/支付账户]
    B --> B3[破坏性批量行为]

    C --> C1[请求频率异常]
    C --> C2[失败率异常]
    C --> C3[消耗突增]
    C --> C4[敏感模型或接口占比异常]

    D --> D1[注册时间]
    D --> D2[充值历史]
    D --> D3[正常使用天数]
    D --> D4[申诉和复核记录]
```

## 五、动作优先级

```mermaid
stateDiagram-v2
    [*] --> low_trust: 新用户
    low_trust --> normal: 正常使用一段时间
    normal --> watched: 轻微异常
    watched --> limited: 重复异常
    limited --> suspended: 达到暂停阈值
    suspended --> normal: 人工恢复
    suspended --> blacklisted: 人工确认高风险
    blacklisted --> [*]
```

建议动作顺序：

1. 先限速，不立即封禁。
2. 再降额度，减少损失面。
3. 多次或高危命中后暂停。
4. 由人工恢复、拉黑或继续观察。
5. 每次动作都写原因码，方便复盘。

## 六、最小落地版本

第一版只做这些：

- 新用户低信任等级。
- 每用户每日额度和请求频率上限。
- 命中硬规则直接拒绝。
- 多次命中后暂停用户。
- 后台能查看事件并手动恢复。

暂不建议第一版做复杂模型评分。先用规则、阈值、审计日志跑出真实样本，再迭代。
