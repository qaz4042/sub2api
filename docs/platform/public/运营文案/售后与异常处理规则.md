# 售后与异常处理规则

更新日期：2026-06-27

适用范围：本站公开 OpenAI API 接入服务的异常处理、费用核查和用户支持。

## 一、处理原则

本站售后处理遵循以下原则：

- 以公开规则为准。
- 以站内记录为依据。
- 能核查的费用问题按记录处理。
- 用户自身配置、代码或密钥管理问题由用户自行承担。
- 上游波动不承诺无限赔付。

## 二、可以申请核查的情况

以下情况可以提交客服核查：

- 请求失败但疑似产生扣费。
- 使用记录和余额变化理解不一致。
- 充值支付成功但余额未到账。
- 同一请求疑似重复扣费。
- 模型价格、倍率或余额口径存在疑问。
- 上游异常导致大面积请求失败。

## 三、通常会补回余额的情况

经核查确认后，以下情况通常会补回对应余额：

- 站内计费逻辑异常导致多扣费。
- 站内重复记录导致同一有效请求重复扣费。
- 支付已成功但站内余额未正确到账。
- 站内配置错误导致用户按错误价格扣费。

## 四、通常不赔付的情况

以下情况通常不属于赔付范围：

- 用户代码循环调用、并发过高或配置错误造成的消耗。
- 用户 API Key 泄露后被他人调用造成的消耗。
- 用户请求上下文过长、输出过多造成的高费用。
- 用户选择高单价模型但未提前确认价格。
- 上游官方接口限流、模型下线或短时波动造成的业务损失。
- 用户将测试 Key 接入生产业务造成的损失。

## 五、请求失败的处理边界

请求失败是否扣费，需要看是否已经形成有效上游消耗。

常见处理方式：

- 未进入上游处理：不应按成功请求扣费。
- 上游明确拒绝且未产生消耗：不应按成功请求扣费。
- 已产生部分输出后流式中断：可能按已产生消耗计费。
- 用户主动中断连接：需结合上游是否已返回消耗记录核查。

最终以站内日志、上游响应、使用记录和账单记录综合判断。

## 六、密钥泄露处理

用户应妥善保管 API Key。

如怀疑泄露，请立即：

1. 删除或禁用相关 API Key。
2. 创建新 Key。
3. 检查近期使用记录。
4. 移除公开仓库、日志、截图、配置文件中的旧 Key。

因用户自身泄露造成的费用消耗，通常不予补偿。

## 七、充值和到账处理

充值或兑换相关问题，请提供：

- 账户标识。
- 支付时间。
- 支付金额。
- 订单号或支付截图。
- 页面显示的错误信息。

如确认支付成功但余额未到账，会按实际支付记录补发站内余额。

## 八、工单信息要求

为了快速处理问题，提交异常时建议提供：

- 账户邮箱或用户 ID。
- 请求时间。
- request_id。
- 使用的模型。
- API Key 名称或备注。
- 错误信息。
- 预期结果和实际结果。

请不要发送完整 API Key、密码、支付密码或其他敏感信息。

## 九、响应时间

建议公开运营时固定客服响应时段，例如：

```text
工作日：10:00 - 22:00
非工作日：不保证实时响应，紧急故障会优先处理
```

具体响应时间以站内公告为准。

## 十、规则更新

如费用口径、上游能力、支付方式或服务范围发生变化，本站可能更新本规则。更新后会在公告中标注更新时间和主要变化。
