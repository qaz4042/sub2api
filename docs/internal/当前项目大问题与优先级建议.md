# 当前项目大问题与优先级建议

检查日期：2026-06-27  
适用项目：Sub2API

## 一、结论

当前项目已经不是普通代理工具，而是“计费型 AI API 网关平台”。README 中明确覆盖鉴权、转发、调度、计费、限流、监控和后台管理，这意味着核心风险不在功能数量，而在稳定性、计费一致性、安全和运维闭环。

建议近期优先处理“能否稳定收费运营”的问题，暂缓继续堆叠非关键功能。

## 二、最高优先级

### 1. 计费一致性与对账闭环

当前风险：

- 项目承诺 Token 级用量统计、余额与倍率计费。
- 扣费路径已有事务和 request_id 去重，这是基础保障。
- 但金额、额度、成本大量使用 `float64`，长期会带来精度和边界问题。
- 请求完成后再记录用量和扣费，仍需明确失败补偿、重复请求、异步队列丢失、余额不足穿透等场景。

建议：

- 建立最小账本模型：请求记录、计费明细、余额变动、订阅用量、API Key 用量应可追溯。
- 明确扣费幂等键、请求指纹和冲突处理规则。
- 为余额不足、上游失败、流式中断、异步记录失败建立补偿策略。
- 增加每日对账任务，检查 usage log、余额扣减、订阅用量、API Key quota 是否一致。

验收标准：

- 任意一笔收费请求都能追溯到请求、上游账号、用户、API Key、模型、Token、倍率和余额变化。
- 重放同一请求不会重复扣费。
- 异步用量记录失败可被发现并补偿。
- 余额和额度不会因为并发请求明显穿透。

### 2. 核心网关路径降复杂度

当前风险：

- `gateway_handler.go`、`openai_gateway_handler.go`、`setting_handler.go` 等文件过大。
- 单个路径中混合请求解析、鉴权、调度、并发、转发、流式错误、计费记录、日志归因。
- 后续改功能时很容易引入隐性回归。

建议：

- 不做一次性大重构，先按关键路径拆小闭环。
- 优先拆出：请求解析、账号选择、上游转发、usage 记录、错误归因、流式处理。
- 每拆一块补最小回归测试，保持行为不变。

验收标准：

- 新增或修改一个模型/上游能力时，不需要同时理解完整 handler 大文件。
- 核心路径测试能覆盖非流式、流式、上游失败、计费失败、并发限流。

## 三、高优先级

### 3. 数据库迁移与升级可靠性

当前风险：

- 迁移文件数量多，且存在同编号多文件。
- 自定义 migration runner 已有 checksum 和 advisory lock，是好的基础。
- 但文档中仍存在 `make migrate-up`、`make migrate-down` 这类当前项目未提供的命令示例。
- 自动启动迁移对生产环境风险较高，尤其在多版本、多实例和大表索引变更时。

建议：

- 修正迁移文档，确保命令真实可用。
- CI 增加空库全量迁移测试。
- 增加旧版本库升级到最新版本的回归测试。
- 对 `_notx.sql`、大表索引、数据回填建立发布检查清单。

验收标准：

- 空库能稳定迁移到最新 schema。
- 代表性旧库能稳定升级。
- 重复启动不会重复执行或卡死迁移。
- 迁移失败时有明确恢复步骤。

### 4. 运行环境版本矩阵收敛

当前风险：

- README 写 PostgreSQL 15+、Redis 7+。
- 默认 Docker Compose 使用 Postgres 18 和 Redis 8。
- 版本支持矩阵不清晰会增加部署和排障成本。

建议：

- 明确生产推荐版本，例如 PostgreSQL 16/17、Redis 7/8。
- Docker Compose 默认版本与推荐版本保持一致。
- CI 或集成测试至少覆盖推荐版本。

验收标准：

- README、部署文档、Compose、CI 对版本要求一致。
- 新用户按默认文档部署不会落入未验证组合。

### 5. 工具链版本策略

当前风险：

- `go.mod` 固定到 `go 1.26.4`。
- CI 也硬校验 `go1.26.4`。
- 这会提高本地开发、CI、发布环境复现成本。

建议：

- 改成团队可稳定获取的 Go 版本策略。
- CI 不要硬编码 patch 版本，除非确实有强依赖。
- 在 README 或开发文档中写清楚本地 Go 版本要求。

验收标准：

- 新开发环境能按文档稳定安装工具链。
- CI 不因 patch 版本获取问题阻塞。

## 四、中优先级

### 6. 前端依赖安全例外处理

当前风险：

- audit exceptions 中存在 high/critical 例外。
- 部分例外将在 2026-07 初到期。
- owner 仍是占位邮箱。

建议：

- 到期前升级或替换相关依赖。
- 对暂时无法处理的例外重新评估影响范围。
- 将 owner 改成真实负责人。

验收标准：

- 没有过期安全例外。
- critical 例外必须有明确缓解措施和复查日期。

### 7. 运营与安全配置默认值

当前风险：

- 项目面向公网收费运营时，默认配置、密钥、URL 白名单、TOTP、备份恢复、日志脱敏都必须形成闭环。
- 现有内部文档已有运营前清单，应继续和代码默认值对齐。

建议：

- 将“运营前配置与数据检查清单”作为上线前必过项。
- 对 URL 白名单、管理员 TOTP、备份恢复、支付回调、日志脱敏做端到端验收。

验收标准：

- 公开运营前有明确 checklist。
- 每个安全项都有可验证结果，而不是只依赖配置说明。

## 五、建议处理顺序

1. 计费一致性与对账闭环。
2. 核心网关路径降复杂度，并补关键回归测试。
3. 数据库迁移可靠性和文档修正。
4. 运行环境版本矩阵与 Go 工具链策略。
5. 前端依赖安全例外清理。
6. 运营前安全配置验收。

## 六、近期不建议优先做

- 继续增加非关键上游能力。
- 大规模 UI 改版。
- 一次性重构全部 handler、service、repository。
- 在未完成计费和迁移可靠性前扩大公开收费运营。
