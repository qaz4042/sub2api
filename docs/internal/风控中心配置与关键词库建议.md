# 风控中心配置与关键词库建议

更新日期：2026-06-29
适用对象：Sub2API 自营实例

## 一、当前结论

当前风控中心已经支持用户级内容风控小闭环：

- 全局或按分组、模型审计用户请求输入。
- 命中关键词、命中历史风险 hash 或命中 moderation 阈值时拦截。
- 写入风控日志。
- 按 `user_id` 累计违规次数。
- 达到阈值后自动禁用用户。
- 后台可查看日志并人工解封。

当前已配置建议：

- 内容审计：开启。
- 模式：前置拦截。
- 审计范围：全部分组、全部模型。
- 自动封禁：开启。
- 封禁阈值：720 小时内命中 3 次。
- 前置 hash 比对：开启。
- cyber_policy：计入封号次数。

## 二、OpenAI API Keys 的作用

风控中心里的 `OpenAI API Keys` 是管理侧审核 Key，不是站内 `api_keys` 表里的用户访问 Key。

它的用途：

- 服务端提取用户请求输入。
- 调用 `{Base URL}/v1/moderations`。
- 使用 `omni-moderation-latest` 等 moderation 模型返回风险分数。
- 与后台阈值比较。
- 决定放行、拦截、记录、累计违规和自动封禁。

注意：

- 站内用户 API Key 只用于识别调用者，不能直接拿来调用 moderation 接口。
- 如果使用第三方兼容服务，必须确认它支持 OpenAI 兼容的 `/v1/moderations`。
- 没有可用审核 Key 时，语义审核不会生效，只能依靠关键词和 hash 拦截。

## 三、成本控制能力

已经实现：

- 采样率：可从 100% 降到 50%、20% 等。
- 按分组审计：可只审计高风险或新用户分组。
- 按模型审计：支持全部、仅包含、排除。
- 关键词本地拦截：命中后不调用 moderation。
- 前置 hash 拦截：历史命中过的相同输入直接拦截。
- 多审核 Key 轮询和异常冻结。

暂未完整实现：

- 按用户风险等级动态采样。
- 新用户全量审计、老用户抽样审计。
- 按单个用户配置审计策略。
- 对上游返回内容做全量二次审核。

推荐省成本顺序：

1. 先配置关键词拦截。
2. 保持前置 hash 比对开启。
3. 必要时把审计范围收窄到高风险分组或模型。
4. 成本仍高时降低采样率。

## 五、配置建议

有审核 Key 时：

- 关键词模式：`关键词 + API`。
- 采样率：先 100%，稳定后按成本调整。
- 阈值：先使用默认阈值，结合日志再微调。

没有审核 Key 时：

- 关键词模式：临时使用 `仅关键词`。
- 保持前置 hash 比对开启。
- 及时补充明显漏拦的关键词。

不建议首批加入：

- 单字词：如 `枪`、`毒`、`杀`。
- 宽泛技术词：如 `破解`、`攻击`、`注入`。
- 新闻、合规咨询中可能大量出现的泛化词。

## 六、运营复盘

建议每周检查一次风控日志：

- 误伤词：删除或改成更精确短语。
- 漏拦表达：补充到关键词库。
- 重复命中输入：依赖 hash 拦截降低成本。
- 高频违规用户：确认是否需要手动禁用或退款限制。
