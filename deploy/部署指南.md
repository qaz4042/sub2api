# Sub2API 部署指南
## 环境
- SSH：`my4g`；目录：`/opt/sub2api`；当前版本：`/opt/sub2api/current`；服务：`sub2api.service`
- 域名：`https://codex.lizubin.online`
- 发布健康检查：远端本机 `http://127.0.0.1:8080/health`

## 部署
```bash
make deploy-my4g
make deploy-my4g-backend-only
```
`deploy-my4g` 前后端一起构建部署；`backend-only` 复用现有前端 dist，仅部署后端。
```bash
# 禁止 dirty 工作区部署
REQUIRE_CLEAN=1 make deploy-my4g
```

## 验证
```bash
ssh my4g 'readlink -f /opt/sub2api/current'
ssh my4g 'systemctl status sub2api --no-pager'
```
公网访问可按需手工验证，不作为发布脚本失败条件：
```bash
curl -fsS https://codex.lizubin.online/health
```

## 调试日志安全
- 生产环境不要开启 `SUB2API_DEBUG_GATEWAY_BODY`；该变量会把网关请求/响应完整 body 写入 `gateway_debug.log`，可能包含 prompt、上传内容和响应内容。
- 临时调试结束后及时删除该变量并重启服务；已有 `gateway_debug.log` 权限应收紧为 `0600`。

```bash
ssh my4g 'sudo find /opt/sub2api -name gateway_debug.log -type f -exec chmod 600 {} \; -exec ls -l {} \;'
```

## 回滚
```bash
ssh my4g 'ls -1dt /opt/sub2api/releases/* | head'
ssh my4g 'sudo ln -sfn /opt/sub2api/releases/<old-release> /opt/sub2api/current && sudo systemctl restart sub2api'
```
部署失败时，脚本会在远端本机健康检查失败后自动回滚到上一版本。
