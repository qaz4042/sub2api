# my2g SSH 连接故障排查与修复

日期：2026-06-21  
目标服务器：`my2g`（`39.102.86.235`）  
涉及客户端：macOS OpenSSH、FinalShell、ClashX Meta

## 一、结论

本次 SSH 故障的主要原因不在阿里云 ECS、安全组、服务器防火墙或 `sshd`，而在本机 ClashX Meta 的 TUN 路由。

ClashX Meta 开启 TUN 后，将发往 `39.102.86.235:22` 的流量送入 `utun4`。连接随后经过代理节点，而代理节点无法正常转发该 SSH 连接，因此客户端在 SSH 协议握手前或握手初期收到：

```text
Connection closed by 39.102.86.235 port 22
kex_exchange_identification: Connection closed by remote host
connection is closed by foreign host
```

最终有效修复是同时完成：

1. 添加目标 IP 的 `DIRECT` 规则。
2. 将目标 IP 加入 TUN 的 `route-exclude-address`，使系统路由直接通过本地网关，而不是先进入 `utun4`。

当前验证结果：

```text
SSH_TUN_EXCLUDE_OK
iZ2ze9vw2u36lpds8csd0fZ
```

## 二、故障现象

### 2.1 命令行 SSH

连接能够建立 TCP，但远端 SSH banner 尚未正常返回时连接被关闭：

```text
debug1: Connecting to 39.102.86.235 port 22
debug1: Connection established
debug1: Local version string SSH-2.0-OpenSSH_9.9
kex_exchange_identification: Connection closed by remote host
```

这表明故障发生在公钥或密码认证之前，不是密钥错误。

### 2.2 FinalShell

FinalShell 使用 `root + 密码` 时显示：

```text
连接主机...
connection is closed by foreign host
```

服务器日志中没有出现对应的密码认证失败记录，因此 FinalShell 的请求当时没有进入服务器的密码验证阶段。

### 2.3 异常端口表现

排查时发现服务器并未监听的 `2022`、`2222`、`22022`，甚至 `65534` 端口也会立即显示 TCP 可连接。

这不是服务器开放了所有端口，而是 TUN 虚拟网络在本机接管了连接。该现象是定位本机代理/TUN 问题的重要证据。

## 四、本机根因证据

### 4.1 故障时的路由

ClashX Meta TUN 开启时：

```text
route to: 39.102.86.235
gateway: 28.0.0.1
interface: utun4
```

ClashX Meta 状态：

```json
{
  "tun": {
    "enable": true,
    "device": "utun4",
    "auto-route": true
  }
}
```

### 4.2 临时关闭 TUN 后的路由

```text
route to: 39.102.86.235
gateway: 192.168.0.1
interface: en1
```

此时 SSH 立即恢复：

```text
SSH_RECOVERED
iZ2ze9vw2u36lpds8csd0fZ
```

### 4.3 为什么单独添加 DIRECT 规则仍不够

已在当前 `魔戒.net` 配置的规则首位加入：

```yaml
rules:
  - IP-CIDR,39.102.86.235/32,DIRECT,no-resolve
```

规则成功加载为第 0 条，但测试时：

```text
hitCount: 0
```

系统路由仍然指向：

```text
gateway: 28.0.0.1
interface: utun4
```

说明在当前 ClashX Meta/TUN 组合下，单独设置 `DIRECT` 规则没有阻止该目标 IP 被 TUN 路由接管。

## 五、实施的修复

### 5.1 配置 DIRECT 规则

当前配置文件：

```text
~/.config/clash.meta/魔戒.net.yaml
```

规则：

```yaml
rules:
  - IP-CIDR,39.102.86.235/32,DIRECT,no-resolve
```

配置已使用当前 Mihomo 核心校验：

```bash
CORE="$HOME/Library/Application Support/com.metacubex.ClashX.meta/.private_core/com.metacubex.ClashX.ProxyConfigHelper.meta"
"$CORE" -t -f "$HOME/.config/clash.meta/魔戒.net.yaml"
```

校验结果：

```text
configuration file ... test is successful
```

### 5.2 配置 TUN 路由排除

通过 ClashX Meta Controller 设置：

```bash
curl -fsS -X PATCH http://127.0.0.1:9090/configs \
  -H 'Content-Type: application/json' \
  -d '{
    "tun": {
      "enable": true,
      "route-exclude-address": [
        "39.102.86.235/32"
      ]
    }
  }'
```

生效后：

```json
{
  "enable": true,
  "device": "utun4",
  "route-exclude-address": [
    "39.102.86.235/32"
  ]
}
```

此时目标 IP 不再走 `utun4`：

```text
gateway: 192.168.0.1
interface: en1
```

SSH 验证成功：

```text
SSH_TUN_EXCLUDE_OK
iZ2ze9vw2u36lpds8csd0fZ
```

## 六、临时恢复方法

如果 SSH 再次失败，可以先临时关闭 TUN：

```bash
curl -fsS -X PATCH http://127.0.0.1:9090/configs \
  -H 'Content-Type: application/json' \
  -d '{"tun":{"enable":false}}'
```

然后验证：

```bash
route -n get 39.102.86.235
ssh my2g
```

正常路由应为本地物理网络接口，如：

```text
gateway: 192.168.0.1
interface: en1
```

## 七、长期配置注意事项

### 7.1 Controller 修改可能不是永久配置

通过 `PATCH /configs` 写入的 `route-exclude-address` 属于当前 Mihomo 运行时配置。以下操作可能使其丢失：

- 重启 ClashX Meta。
- 切换订阅。
- 更新当前订阅。
- ClashX Meta 自动恢复 TUN。
- 重新加载未包含排除项的配置。

因此，仅依赖运行时 API 不够稳定。

### 7.2 推荐持久化方式

应优先在 ClashX Meta 的覆写、Mixin 或 TUN 高级配置中持久化：

```yaml
tun:
  route-exclude-address:
    - 39.102.86.235/32
```

同时保留：

```yaml
rules:
  - IP-CIDR,39.102.86.235/32,DIRECT,no-resolve
```

如果直接修改远程订阅生成的 YAML，订阅更新可能覆盖修改，应使用客户端提供的覆写机制。

## 八、FinalShell 配置要求

FinalShell 应配置为：

```text
主机：39.102.86.235
端口：22
用户名：root
认证方式：密码或正确的私钥
代理服务器：无
跳板机：无
SSH 隧道：关闭
```

注意：

- 不要使用业务域名作为 SSH 主机。
- `.env` 中的 `ADMIN_PASSWORD` 是 Sub2API 管理后台密码，不是服务器 root 密码。
- 如果 FinalShell 单独配置了 HTTP/SOCKS5/系统代理，即使系统路由已经直连，也可能继续通过代理连接。
- 如果命令行 `ssh my2g` 成功而 FinalShell 失败，优先检查 FinalShell 自身的代理、跳板机和缓存连接配置。

## 九、快速诊断命令

### 9.1 检查本机路由

```bash
route -n get 39.102.86.235
```

判定：

```text
interface: utun4  -> 仍被 TUN 接管
interface: en0/en1 -> 正在通过本地网络直连
```

### 9.2 检查 TUN 状态

```bash
curl -sS http://127.0.0.1:9090/configs |
  jq '.tun | {enable,device,"route-exclude-address":."route-exclude-address"}'
```

### 9.3 检查规则是否加载

```bash
curl -sS http://127.0.0.1:9090/rules |
  jq -r '.rules[] |
    select(.payload=="39.102.86.235/32") |
    [.index,.type,.payload,.proxy,.extra.hitCount] |
    @tsv'
```

### 9.4 详细测试 SSH

```bash
ssh -vv -o ConnectTimeout=10 my2g
```

### 9.5 检查服务器 SSH 日志

```bash
ssh my2g \
  'journalctl -u sshd --since "-10 min" --no-pager | tail -100'
```

如果客户端报错，但服务器日志完全没有对应连接，优先排查本机代理、TUN、FinalShell 代理配置或中间网络。

## 十、判断流程

```text
SSH 连接失败
  |
  +-- route 显示 utun4
  |     |
  |     +-- 临时关闭 TUN
  |     +-- 配置 route-exclude-address
  |     +-- 保留目标 IP 的 DIRECT 规则
  |
  +-- route 显示 en0/en1
        |
        +-- 命令行 SSH 成功、FinalShell 失败
        |     -> 检查 FinalShell 代理/跳板机/主机配置
        |
        +-- 命令行 SSH 也失败
              -> 检查阿里云安全组、sshd、防火墙和服务端日志
```

## 十一、安全建议

当前服务器允许 root 密码登录：

```text
PermitRootLogin yes
PasswordAuthentication yes
```

为了减少公网暴力破解风险，后续确认密钥登录稳定后，建议：

1. 创建独立运维用户并授予必要的 `sudo` 权限。
2. 优先使用 ED25519 密钥。
3. 将安全组的 22 端口来源限制为可信公网 IP。
4. 再评估关闭 root 密码登录。

在密钥登录和备用控制台入口验证完成前，不要直接关闭现有认证方式。
